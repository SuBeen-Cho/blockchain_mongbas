## 문서 개요

이 문서는 BFT 기반 익명 전자투표 시스템(Project Mongbas)에 대해 실제로 발생 가능한 5가지 보안 위협 시나리오를 상정하고, 각 시나리오별로 기존 측정에서 누락된 항목을 추가 측정한 결과입니다.

각 시나리오는 다음 구조로 기술됩니다:
1. **위협 시나리오** — 어떤 공격자가 어떤 상황에서 공격을 시도하는가
2. **공격자 행동** — 구체적으로 어떤 방법으로 공격하는가
3. **시스템 대응** — 시스템이 이를 어떻게 막는가
4. **측정 결과** — 방어가 실제로 동작하는지 수치로 검증

## 시나리오 A — 선관위 단독 결과 조작

### 위협 시나리오

선거관리위원회(EC) 내부자가 자신의 조직 피어(`peer0.ec`)에 직접 접근하여 특정 후보의 득표수를 임의로 조작하거나, 합법적인 선거 종료 시점을 앞당겨 결과를 유리하게 만들려 시도합니다. 전자투표 시스템에서 단일 관리 주체가 결과를 통제할 수 있다면 선거의 정당성 자체가 붕괴됩니다.

### 공격자 행동

- EC 조직의 admin 키로 `CloseElection` 트랜잭션을 단독 제출
- `peer0.ec` 서명만 포함된 트랜잭션으로 원장 상태 변경 시도
- 참관 기관(PartyObserver, CivilSociety)의 동의 없이 결과 확정 시도

### 시스템 대응

Hyperledger Fabric 채널 승인 정책으로 `OutOf(2, EC.peer, Party.peer, Civil.peer)` 를 강제합니다. 이 정책은 3개 기관 중 반드시 2개 이상의 서명이 있어야 트랜잭션이 유효하다는 것을 의미하며, 선관위 단독 서명 트랜잭션은 Endorsement 단계에서 수학적으로 거부됩니다. Fabric Gateway가 이 정책 충족을 자동으로 검증하므로 코드 레벨 추가 방어 없이도 보장됩니다.

## 시나리오 B — 이중투표 시도

### 위협 시나리오

동일한 유권자가 같은 선거에 두 번 투표하여 특정 후보의 득표수를 인위적으로 늘리려 합니다. 온라인 투표 시스템에서 이중투표 방지는 가장 기본적이고 핵심적인 보안 요건입니다. 특히 익명성을 보장하면서 동시에 이중투표를 막는 것은 기술적으로 까다로운 문제입니다.

### 공격자 행동

- 동일한 `voterSecret`으로 `CastVote`를 반복 호출
- 동일한 `nullifierHash`로 다른 `candidateID`에 재투표하여 원하는 후보 득표 증가 시도
- 또는 `nullifierHash`를 조작하여 새로운 투표인 것처럼 위장 시도

### 시스템 대응

`nullifierHash = SHA256(voterSecret + electionID)`로 계산되므로 동일한 `voterSecret`은 항상 동일한 `nullifierHash`를 생성합니다. 체인코드 내 `CastVote`는 Nullifier Eviction 방식을 채택하여, 동일한 nullifier로 재투표 시 기존 투표를 덮어씁니다. 이로써 마지막 투표 1개만 집계에 반영되어 이중집계가 원천 불가능합니다. SHA256의 충돌 저항성이 nullifier 유일성을 수학적으로 보장합니다.

---

### B-2 — SHA256 해시 충돌 저항성 측정 (10만 개)

#### 측정 내용
서로 다른 `voterSecret` 10만 개로 `nullifierHash`를 생성하고, 동일한 해시값이 두 번 이상 등장하는 충돌이 발생하는지 측정합니다. 충돌이 존재하면 두 유권자가 동일한 nullifier를 가지게 되어, 한 명의 투표가 다른 명의 투표를 덮어쓰는 부정이 가능해집니다.

#### 측정 결과

| 항목 | 값 |
|------|----|
| 생성된 nullifierHash 수 | 100,000개 |
| **충돌 건수** | **0건** |
| **충돌률** | **0.000000%** |
| 측정 소요 시간 | 103ms |
| 처리 속도 | 970,874개/초 |

#### 시사점
- SHA256 출력 공간은 2²⁵⁶ ≈ 1.16 × 10⁷⁷ — 10만 개 수준에서 충돌은 수학적으로 사실상 불가능
- 충돌 **0건** — nullifier 유일성이 완전 보장됨
- Birthday paradox 기준 50% 충돌 확률에 필요한 샘플 수는 약 2¹²⁸ ≈ 3.4 × 10³⁸개 → 현실적 공격 불가
- 전국 단위 선거(유권자 약 4,400만 명)의 10,000배 규모에서도 충돌 없음

---

### B-3 — Eviction 오버헤드 측정 (50회)

#### 측정 내용
정상 CastVote와 동일 nullifier로 재투표(Eviction)하는 경우의 레이턴시를 각 50회씩 비교합니다. 오버헤드가 크다면 공격자가 대량 Eviction을 이용한 DoS(서비스 거부) 공격에 악용할 수 있습니다.

#### 측정 결과

| 구분 | avg | stddev | min | max | p50 | p95 | p99 |
|------|-----|--------|-----|-----|-----|-----|-----|
| 정상 CastVote | 615.5ms | 35.9ms | 580ms | 758ms | 606ms | 703ms | 758ms |
| Eviction (재투표) | 620.5ms | 51.6ms | 573ms | 904ms | 603ms | 682ms | 904ms |

| 오버헤드 | t-통계량 | 임계값 (p=0.05) | 통계적 유의성 |
|---------|---------|----------------|-------------|
| **5.0ms** | 0.559 | 2.010 | ✅ 통계적 차이 없음 |

#### 시사점
- Eviction 처리 오버헤드 **5.0ms** — 목표 10ms 미만 달성 ✅
- t=0.559 < 2.010, p > 0.05 — 정상 투표와 Eviction 투표의 처리 비용이 **통계적으로 동일**
- 재투표가 정상 투표와 동일한 처리 경로를 거치므로 별도 오버헤드 없음
- 공격자가 동일 nullifier로 수천 번 재투표를 시도해도 **특별한 취약점(DoS 증폭 등)이 없음**

## 시나리오 C — 강압 투표 (협박자 검증 요구)

### 위협 시나리오

협박자(강압자)가 유권자에게 "누구에게 투표했는지 증명하라"고 강요합니다. 유권자는 실제로 A 후보에게 투표했지만, 협박자의 요구에 응하면서 동시에 자신의 실제 투표 의사를 보호해야 합니다. 선거의 자유 원칙 중 '비밀 투표'는 강압에 굴복하지 않을 자유를 포함합니다.

### 공격자 행동

- 유권자의 `voterSecret`을 직접 빼앗아 자신이 직접 Merkle 증명을 확인하려 시도
- 또는 유권자에게 증명 API를 실행하도록 강요하고 그 결과를 지켜봄
- 협박자가 응답을 모니터링하면서 `normalPassword`와 `panicPassword` 중 어느 것을 사용했는지 **응답 시간이나 응답 구조**로 추측 시도

### 시스템 대응

`GetMerkleProofWithPassword(nullifierHash, passwordHash)` API는 두 가지 경로를 가집니다:
- `normalPasswordHash` 입력 → 실제 투표 증명 반환
- `panicPasswordHash` 입력 → 더미 후보에게 투표한 가짜 증명 반환

두 경우 모두 응답 구조, 데이터 길이, 처리 시간이 동일하도록 설계되어 협박자가 어떤 방법으로도 Normal/Panic을 구별할 수 없습니다.

---

- 200회 확대 측정에서도 **타이밍 차이가 통계적으로 유의미하지 않음** → 재현성 확인

- 협박자가 타이밍 기반 side-channel 공격으로 Normal/Panic을 구별하는 것은 **200회 측정 기준으로도 불가**
- 샘플을 늘려도 판정이 동일 → 방어 메커니즘의 통계적 견고성 입증

### C-3 — Normal/Panic 응답 구조 동일성 검증 (50회)

#### 측정 내용
Normal과 Panic 응답의 JSON 필드 구조 및 응답 바이트 길이를 50회 자동 비교합니다. 협박자가 응답을 캡처해서 필드 수, 필드명, 응답 크기를 분석하면 어느 비밀번호를 사용했는지 구별할 수 있으므로, 이 채널도 차단되어야 합니다.

#### 측정 결과

| 항목 | 값 |
|------|----|
| 총 검증 횟수 | 50회 |
| **JSON 필드 구조 동일** | **50/50 = 100.0%** |
| **응답 길이 완전 동일 (0바이트 차이)** | **50/50 = 100.0%** |
| 구조 불일치 횟수 | 0회 |

#### 응답 바이트 길이 차이 분포

| avg | stddev | min | max | p50 | p95 | p99 |
|-----|--------|-----|-----|-----|-----|-----|
| 0.0bytes | 0.0bytes | 0bytes | 0bytes | 0bytes | 0bytes | 0bytes |

#### 시사점
- JSON 필드 구조 동일률 **100.0%** → 협박자가 필드 구조 분석으로 구별 불가
- 응답 바이트 길이 차이 **0bytes** → 패킷 크기 분석으로도 구별 불가

- 협박자가 외부에서 관찰 가능한 어떤 채널을 통해서도 Normal/Panic을 구별할 수 없음

## 시나리오 D — 집계 키 단독 탈취

### 위협 시나리오

참관 정당 서버가 외부 해커에게 침해되어 Shamir Secret Share 1개가 탈취됩니다. 공격자는 이 Share 1개로 선거 결과 집계에 사용되는 masterKey를 복원하려 합니다. masterKey가 복원되면 모든 암호화된 투표 데이터를 복호화할 수 있어 투표 비밀이 전면 노출됩니다.

### 공격자 행동

- 탈취한 Share 1개로 `SubmitKeyShare`를 단독 호출하여 masterKey 복원 시도
- 또는 위조된 Share를 만들어 2-of-3 조건을 흉내 내어 복원 시도
- 복원 실패/성공 여부를 응답 시간으로 추측하는 side-channel 공격 시도

### 시스템 대응

GF(p) 소수체(p = secp256k1 prime, 2^256 - 2^32 - 977) 위에서 `f(x) = masterKey + coeff·x mod p` 형태의 1차 다항식으로 Share를 분배합니다. 32바이트 masterKey 전체를 하나의 256비트 정수로 처리하므로 보안 공간이 약 2^256입니다. Share 1개만으로는 수학적으로 무한히 많은 masterKey가 가능하여 실질 정보량이 0입니다. `SubmitKeyShare` 체인코드는 Share 제출 횟수가 threshold(2)에 미달하면 복원을 실행하지 않으며, Feldman VSS 커밋먼트 + SHA256(복원값) == keyHash 검증으로 위조 Share도 즉시 차단합니다.

#### 측정 내용
Share 1개 제출(복원 실패) vs Share 2개 제출(복원 성공) 시나리오 각 50회의 응답 시간을 비교합니다. 성공 시 masterKey 복원 및 SHA256 검증 과정이 추가되므로 처리 시간이 길어질 수 있으며, 이 차이를 공격자가 측정하면 "내 Share가 올바른 것인지" "threshold에 얼마나 근접했는지"를 추측하는 side-channel로 악용될 수 있습니다.

## 시나리오 E — "결과가 조작됐다" 외부 주장

### 위협 시나리오

선거에서 패배한 측이 "최종 집계 결과가 블록체인 기록과 다르다", "내 투표가 반영되지 않았다", "Merkle Tree가 선거 후에 조작됐다"고 주장합니다. 민주적 선거에서 결과에 대한 불복과 의혹 제기는 매우 빈번하며, 시스템은 이에 대해 수학적으로 반박할 수 있는 증거를 제공해야 합니다.

### 공격자 행동

- 가짜 `nullifierHash`로 Merkle Proof를 요청하여 "포함됐다"는 허위 응답을 유도하려 시도
- BuildMerkleTree 완료 후 추가 투표를 삽입하여 Root Hash를 변경하려 시도
- 대규모 선거에서 Merkle Proof 요청을 폭주시켜 검증 서비스를 마비시키려 시도

### 시스템 대응

Merkle Tree는 `CloseElection` 이후 `BuildMerkleTree`가 호출된 시점에 단 한 번 원장에 기록되며, 이후 수정이 불가능합니다. 존재하지 않는 nullifier에 대한 Proof 요청은 `included: false`를 반환하여 허위 포함 주장을 즉시 반박합니다. O(log N) 알고리즘으로 대규모 선거에서도 빠른 증명이 가능합니다.

## 시나리오 F — BatchTimeout 변경에 따른 성능·보안 트레이드오프

### 위협 시나리오

악의적인 오더러 운영자 또는 내부 공모 기관이 채널의 `BatchTimeout` 파라미터를 임의로 변경하여 선거를 방해하려 시도합니다. 구체적인 공격 형태는 두 가지입니다:

1. **고(高) BatchTimeout 공격**: BatchTimeout을 5s~30s로 높게 설정하여 트랜잭션 레이턴시를 급증시킴으로써 유권자가 투표 확인을 오래 기다리게 만들어 선거 혼란을 유발합니다.
2. **저(低) BatchTimeout 공격**: BatchTimeout을 100ms 이하로 낮게 설정하여 블록 생성 빈도를 폭증시키고 오더러/피어에 부하를 가중시키는 DoS 공격을 시도합니다.

### 공격자 행동

- `peer channel update` 명령으로 채널 config 업데이트를 시도
- BatchTimeout을 현재 2s에서 500ms 또는 5s로 변경하려 시도
- 단일 기관 Admin 서명만으로 채널 config 변경 시도

### 시스템 대응

채널 config 업데이트는 `Orderer/Admins` 그룹 정책에 따라 **OrdererMSP Admin + 3개 피어 조직 Admin 전체 서명**이 요구됩니다. 따라서:

- 단일 기관이 BatchTimeout을 임의로 변경하는 것은 불가능
- 변경이 이루어지더라도 BatchTimeout은 레이턴시에 비례하여 영향 → 성능 측정으로 정량화 가능
- 실제 선거 운영에서 BatchTimeout 조작이 어느 정도의 성능 영향을 미치는지 측정하여 위협의 심각도를 정량화

### F-1 — BatchTimeout × TPS 교차 측정 (실제 채널 config 변경)

#### 측정 내용

`update-batchtimeout.sh`를 이용하여 실제 채널 config를 온라인으로 변경하고(네트워크 재시작 없음), 각 BatchTimeout 값에서 TPS 1/3/5/10/20을 측정합니다.

#### 측정 조건

| 항목 | 값 |
|------|-----|
| BatchTimeout 종류 | 500ms / 1s / 2s(기본) / 5s |
| TPS 목표 | 1 / 3 / 5 / 10 / 20 |

#### 분석 및 시사점

**1. BatchTimeout은 직렬 환경에서 레이턴시를 선형적으로 지배한다**

본 측정에서 실제 TPS는 목표 TPS와 무관하게 BatchTimeout에만 비례합니다. BT=500ms에서 실제 TPS≈1.5, BT=5s에서 실제 TPS≈0.19로, BatchTimeout이 10배 증가하면 처리량이 ~8배 감소했습니다. 이는 직렬(1-worker) 환경에서 병목이 순전히 BatchTimeout에 의한 블록 대기임을 뜻합니다.

**2. 고TPS 환경에서도 MaxMessageCount(500) 지배로의 전환이 관찰되지 않았다**

이론적으로 TPS가 높아 MaxMessageCount(500)에 빠르게 도달하면 BatchTimeout과 무관한 레이턴시를 보여야 하나, 직렬 1-worker 구조상 누적 미처리 트랜잭션이 500개에 달하지 않아 항상 BT 지배 구간에 머물렀습니다. MaxMessageCount 지배 구간을 관측하려면 다수의 병렬 워커가 필요합니다.

**3. BatchTimeout 조작은 선거 응답성을 정량적으로 저하시킨다**

악의적으로 BT=5s로 설정된 경우, TPS=1 환경에서 avg 5240ms(~5.2초)의 레이턴시가 발생합니다. 실제 선거에서 유권자가 투표 확인까지 5초 이상 대기해야 한다면 심각한 사용자 경험 저하 및 시스템 신뢰 손상이 발생합니다.

**4. 채널 config 업데이트 정책으로 단독 조작 불가**

BatchTimeout 변경은 `OrdererMSP Admin + ElectionCommissionMSP + PartyObserverMSP + CivilSocietyMSP` 4개 서명이 모두 필요하므로, 단일 기관이 이를 임의로 변경할 수 없습니다. 즉, BatchTimeout 조작 위협은 **다기관 공모 없이는 실행 불가**합니다.

**5. 실패율 0% — 모든 BatchTimeout 설정에서 데이터 무결성 유지**

BT=500ms~5s 전 구간에서 실패율 0.0%. 레이턴시는 변해도 투표 데이터의 정확성/무결성은 훼손되지 않음을 확인했습니다.

## 시나리오 G — 동시 투표 인원 증가에 따른 병목 판단

### 위협 시나리오

선거 당일 투표 시작 직후 대규모 유권자가 동시에 투표하는 상황. 시스템이 고동시 부하를 처리하지 못하면 투표 창구 마비(서비스 거부 효과)가 발생하거나 처리량(TPS)이 조기에 포화될 수 있습니다. 이 측정은 동시 투표 수를 단계적으로 증가시키면서 레이턴시·TPS·실패율·병목 구간을 정량 분석합니다.
