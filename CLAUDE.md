## Project Overview

BFT-based anonymous e-voting system built on Hyperledger Fabric 2.5. Core privacy mechanisms: Nullifier-based anonymous voting, Private Data Collections (PDC), Shamir Secret Sharing (2-of-3 threshold decryption), and deniable Merkle proofs with Panic Password support.

**Endorsement Policy:** 2-of-3 multi-org (ElectionCommission, PartyObserver, CivilSociety) — no single org can manipulate results.

## Network Management

```bash
cd network/
./scripts/network.sh up        # cryptogen + configtxgen + docker compose + channel join
./scripts/network.sh deploy    # install chaincode on all peers + approve + commit + InitLedger
./scripts/network.sh test      # smoke test: CastVote + double-vote check + eviction
./scripts/network.sh down      # stop containers + delete volumes
./scripts/network.sh clean     # full reset including crypto-config and channel-artifacts
```

`crypto-config/` and `channel-artifacts/` are gitignored — always regenerated by `network.sh up`. Chaincode sequence auto-increments on each deploy (currently at 9).

## Backend API Server

```bash
cd application/
npm install
npm start          # Express server on port 3000
npm run dev        # Dev mode with nodemon
```

Health check: `GET /health` — also returns Idemix auth status.

Environment variables:
- `PORT` — defaults to 3000
- `IDEMIX_ENABLED=true/false` — toggle Idemix auth (dev toggle in `middleware/auth.js`)
- `SESSION_SECRET` — session signing key
- `PANIC_PASSWORD` — enable deniable proof mode

## Chaincode (Go)

The smart contract is in `chaincode/voting/voting.go` (single file, ~1400 lines). To modify and redeploy:

```bash
# After editing chaincode/voting/voting.go:
cd network/
./scripts/network.sh deploy    # auto-increments sequence
```

No separate Go build step needed — `network.sh deploy` handles packaging and CCAAS deployment.

## Architecture

### Data Flow for Vote Submission

```
Browser
  ├── voterSecret  (stays in browser, never sent)
  └── nullifierHash = SHA256(voterSecret + electionID)  [crypto.js]
        │
        ├── Transient map → PDC (VotePrivate, Shamir shares) — bypasses orderers
        └── Public ledger → Nullifier { nullifierHash, candidateID, evictCount }
```

### Component Map

| Layer | Location | Key Files |
|-------|----------|-----------|
| Blockchain | `chaincode/voting/` | `voting.go`, `collection_config.json` |
| Network | `network/` | `docker-compose.yaml`, `configtx.yaml`, `crypto-config.yaml` |
| Backend | `application/src/` | `app.js`, `gateway.js`, `routes/`, `middleware/auth.js` |
| Frontend | `frontend/src/` | `pages/` (Voter, Admin, Verify), `utils/crypto.js` |
| Benchmarks | `caliper/`, `scripts/` | `run-caliper.sh`, `bench_full.sh` |

### Fabric Gateway (application/src/gateway.js)
- Connects to `peer0.ec` (localhost:7051) as `ElectionCommissionMSP`
- Channel: `voting-channel`, Chaincode: `voting`
- Timeouts: evaluate 5s, endorse 15s, commit 60s

- peer1.ec: http://localhost:5985/_utils

## Chaincode Functions Reference

The 16 chaincode functions follow a strict election lifecycle:
`CREATED → ACTIVE → CLOSED → (Merkle built) → (Shamir restored)`

Key functions: `CreateElection`, `ActivateElection`, `CastVote` (with Idemix + PDC + Eviction), `CloseElection` (auto-tallies via CouchDB rich query), `BuildMerkleTree`, `GetMerkleProofWithPassword` (Normal/Panic branching), `InitKeySharing`, `SubmitKeyShare` (auto-restores masterKey when n≥2).
